# Análisis y Diseño de Software

# clase número 1

**Objetivo**
Entender el ciclo de vida del software y comparar metodologías ágiles vs estructuradas.

**¿Por qué el análisis y diseño son importantes?**

Hay un ´66%´ de los proyectos fracasados, ya que muchas de las personas comienzan a programar sin antes haber pensado en el diseño, la precentación, los gustos del cliente, etc.

**¿Qué es el análisis?**

Este trata de **entender el problema** antes de comenzar.

- ¿Quién va a usar el sistema?
- ¿Qué necesita exactamente?
-¿Qué información maneja?
-¿Qué restricciones existen (presupuesto, tiempo, leyes)?

**¿Qué es el diseño?**

Diseño significa **planear la solución** antes de construirla. Es responder:

-¿Cómo se va a organizar el sistema?
-¿Qué partes lo componen y cómo se comunican?
-¿Qué tecnologías se usarán?
-¿Cómo se va a ver y sentir para el usuario?

# El ciclo de vida del software

El ciclo de vida del software es el conjunto de etapas por las que pasa un sistema desde que es una idea hasta que se vuelve obsoleto y deja de usarse. Como un ser vivo: nace, crece, madura, se reproduce (versiones) y eventualmente muere.

**Análisis de Requerimientos**
Se conoce las necesidades de los clientes, tomando nota de cada requerimiento.  Es la fase MÁS importante: si aquí algo sale mal, todo lo demás también.

**Diseño**
Se decide la arquitectura: qué módulos habrá, cómo se conectan, qué base de datos se usará, cómo se verá la interfaz.

**Implementación (Programación)**
Los programadores escriben el código siguiendo los diseños.

**Pruebas**
Se verifica que el software haga lo que debe y no tenga errores.

**Despliegue**
Se entrega el software al cliente y se pone en funcionamiento real.

**Mantenimiento**
Se corrigen errores que aparecen con el uso y se agregan nuevas funciones.

Las metodologías estructuradas se basan en una idea: planificar todo desde el principio y seguir el plan. Son como construir un puente — necesitas saber TODO antes de empezar a soldar.

#  Metodologías Ágiles

**La revolución ágil**
Creación el Manifiesto Ágil: una nueva forma de pensar el desarrollo de software.

**El Manifiesto Ágil**
-Individuos e interacciones
-Software funcionando
-Colaboración con el cliente
-Responder al cambio

**Scrum**:es una metodología ágil que divide el trabajo en **sprints** de 1 a 4 semanas.
- **Product Owner:** decide qué se desarrolla.
- **Scrum Master:** apoya al equipo y elimina obstáculos.
- **Equipo de desarrollo:** construye el producto.
- **Daily Standup:** reunión diaria breve para revisar avances.
- **Demo:** presentación del trabajo realizado.
- **Retrospectiva:** análisis para mejorar el proceso.

**Kanban**: tablero con columnas.
**XP (Extreme Programming)**: programación en pares,Test Driven Development,Refactorización constante,Releases pequeños y frecuentes.

# Lectura dirigida: ¿Qué es un requerimiento?

Condiciones del cliente traducidas al lenguaje de los desarrolladores.

**Tipos de requerimientos**
- Funcionales:Describen qué debe hacer el sistema. Son las acciones, las funcionalidades concretas.
- No funcionales: Describen cómo debe ser el sistema. Características de calidad y restricciones.

**Buenos vs malos requerimientos**
Cumplir 5 cualidades. En inglés se conocen como SMART.
**S**pecific-Específico
**M**easurable-Medible
**A**chievable—Alcanzable
**R**elevant—Relevante
**T**ime-bound—Con plazo

**¿De dónde salen los requerimientos?**
 técnicas de elicitación. Esta ayuda a sacar información de: Entrevistas, encuestas, observación, talleres, análisis de documentos y prototipos.

**¿Pueden cambiar los requerimientos?**
Estructurados: Los minimizan documentando todo al inicio.
Ágiles: Los aceptan y se adaptan a ellos.
Ingeniero: No dice "no se puede"; calcula costos y riesgos y pregunta: ¿lo aprobamos?
